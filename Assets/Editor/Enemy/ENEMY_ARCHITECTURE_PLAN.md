# План архитектуры врагов

Документ фиксирует принятые решения по первой итерации системы врагов.

## Цели

- Сделать первых реальных противников без захардкоженного поведения в одном компоненте.
- Подготовить основу под разные AI-архетипы для платформера.
- Дать удобную базу для будущего `Enemy Editor`.
- Не ломать текущий `EnemySpawner`, `EnemyEntity`, `EnemyHealth`.

## Главные слои системы

### 1. Данные врага

Главный asset врага: `EnemyDataSO`.

Он должен хранить:

- `ID`
- `DisplayName`
- `Prefab`
- `AIType`
- `Stats`
- `Perception`
- `Movement`
- `Attack`
- `Rewards`

Старое поле `BaseStats` оставляем временно как legacy-слой для обратной совместимости.

### 2. Рантайм-сущность

`EnemyEntity` остаётся точкой сборки.

Внутри префаба врага дальше должны жить отдельные компоненты:

- `EnemyStats`
- `EnemyHealth`
- `EnemyBrain`
- `EnemySensor2D`
- `EnemyLocomotion2D`
- `EnemyAttackController`
- `EnemyAnimationBridge`

### 3. Поведение

На текущий момент фиксируем 4 основных архетипа AI:

1. `AgileJumper`
   - быстрый преследователь
   - умеет активно перепрыгивать платформы
   - может резко сокращать дистанцию до игрока

2. `GroundChaser`
   - обычный мили-преследователь
   - идёт к игроку по платформе
   - атакует вблизи
   - не пытается войти в игрока
   - базовый пример: зомби

3. `StaticCaster`
   - статичный враг
   - имеет большой радиус обнаружения
   - атакует дистанционно
   - не пытается reposition / убегать

4. `KitingRanged`
   - стрелок
   - держит дистанцию
   - отступает при сближении игрока
   - при необходимости может использовать перепрыгивание

Архетип `Patrol` в текущий план не входит.

## Платформенная логика

### Базовый уровень

Для первой волны врагов нужны:

- проверка земли под ногами
- понимание края платформы
- понимание стены перед собой
- остановка перед игроком на `StopDistance`
- атака только в допустимом `AttackRange`

### Расширение позже

Автоматический pathfinding по платформам пока не делаем.

Вместо этого во второй фазе вводим `EnemyJumpLink`:

- заранее расставленные точки перехода между платформами
- AI выбирает подходящий link вместо попытки "самому понять карту"

Это проще, надёжнее и лучше контролируется в platformer-уровнях.

## Статы и скейл

Для новых врагов используем записи вида:

- `StatType`
- `BaseValue`
- `ScalingMode`
- `ScalingValue`

Поддерживаем режимы:

- `None`
- `FlatPerLevel`
- `PercentPerLevel`

Это даёт контроль над:

- HP
- уроном
- скоростью
- бронёй
- уклонением
- резистами

## Урон и сопротивления

У врага должна быть настраиваемая атака по типу урона:

- Physical
- Fire
- Cold
- Lightning

И редактируемые сопротивления:

- `PhysicalResist`
- `FireResist`
- `ColdResist`
- `LightningResist`

Они живут в статах врага и должны быть доступны в будущем `Enemy Editor`.

## Enemy Editor

`Enemy Editor` будет отдельным окном, а не просто инспектором.

Что в нём должно быть:

- список врагов
- поиск
- фильтр по `AIType`
- редактирование `EnemyDataSO`
- редактирование статов и скейлов
- редактирование атаки, типа урона и movement/perception параметров
- preview итоговых статов на выбранном уровне
- валидация префаба врага

Что не входит в обязанности Enemy Editor:

- монтаж анимаций
- редактирование Animator Controller
- покадровая настройка hitbox-анимаций

## Фазы реализации

### Фаза 1. Data contract

- расширить `EnemyDataSO`
- перевести `EnemySpawner` на префаб из `EnemyDataSO`
- обновить `EnemyStats` под новый формат статов и скейлинга

### Фаза 2. Runtime foundation

- `EnemySensor2D`
- `EnemyLocomotion2D`
- `EnemyBrain`
- `EnemyAttackController`
- `EnemyAnimationBridge`

### Фаза 3. Первый реальный враг: Zombie

- idle / move / attack
- `GroundChaser`
- stop distance
- мили-атака
- понимание края платформы

### Фаза 4. Enemy Editor

- окно редактора врагов
- фильтры
- preview
- validation

### Фаза 5. Jump links

- `EnemyJumpLink`
- использование связей между платформами

### Фаза 6. Расширения

- ranged AI
- caster AI
- elite-модификаторы
- особые механики врагов
